# API Documentation Policy

The SDK has two API documentation layers:

- TypeDoc, generated from the wrapper entrypoint and synced resource
  modules into `docs/api/`.
- Compact per-resource markdown, generated from synced Fern output into
  `wrapper/docs/resources/`.

Both are generated documentation surfaces. They should be reproducible,
package-name-correct, and connected to the same OpenAPI truth as the SDK
itself.

## Rules

| Surface | Rule |
|---|---|
| TypeDoc config | Entry points stay `index.ts` and `src/api/resources`, output stays `../docs/api`, and package naming uses `clockify-sdk-ts-115`. |
| Resource docs | Generated by `wrapper/scripts/gen-resource-docs.ts` from `wrapper/src/api/resources`; do not hand-edit when SDK shape changes. |
| Sync path | `npm run sync` must regenerate resource docs after copying Fern output. |
| Published docs | GitHub Pages workflow builds TypeDoc with `npm run docs`; generated `docs/api/` is not a source-of-truth input. |
| Package docs | `wrapper/README.md` should link users to reference docs and resource docs without contradicting package names. |
| Generated HTML identity | If `docs/api/` exists in the checkout, its TypeDoc title, package name, and npm navigation link must match `clockify-sdk-ts-115`; stale `clockify-sdk-ts` pages must be regenerated, not hand-edited. |
## Required receipts

Before claiming API-doc readiness, run or cite:

- `make api-docs`
- `cd wrapper && npm run docs`
- `cd wrapper && npm run docs:resources`
- `make user-docs`
- `make docs-index-drift`
- generated HTML identity check when `docs/api/` exists

Do not use generated API pages to justify behavior that is not backed
by the corrected OpenAPI snapshot or wrapper tests.
